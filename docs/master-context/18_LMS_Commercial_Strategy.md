# 18. LMS Commercial Strategy

Статус: коммерческая стратегия для MVP, первых клиентов и дальнейшего роста  
Продукт: корпоративная LMS / Learning Operating System  
Язык продукта: русский

---

## 1. Назначение документа

Этот документ описывает коммерческое позиционирование LMS: для кого продукт, какую проблему решает, как его продавать, какие функции важны для первых клиентов и как развивать продукт после MVP.

Документ нужен для:

- выбора целевого сегмента;
- упаковки MVP;
- подготовки первых продаж;
- приоритизации функций;
- понимания отличий от конкурентов;
- формирования pricing model;
- подготовки future enterprise roadmap.

---

## 2. Главная продуктовая идея

Продукт должен позиционироваться не как “ещё одна LMS”, а как:

> корпоративная система управления обучением, которая помогает быстро запускать onboarding, compliance training и развитие сотрудников без тяжёлой enterprise-внедренческой сложности.

Ключевой фокус:

- простота запуска;
- корпоративные роли;
- обучение сотрудников;
- контроль выполнения;
- отчётность;
- сертификаты;
- готовность к росту;
- будущая AI-автоматизация.

---

## 3. Целевые сегменты

### 3.1 Малый и средний бизнес

Потребности:

- быстро обучать сотрудников;
- стандартизировать onboarding;
- хранить учебные материалы;
- видеть прогресс;
- не платить за тяжёлые enterprise LMS.

Преимущество продукта:

- простота;
- быстрый запуск;
- русский интерфейс;
- понятная админка;
- Railway/SaaS-first или self-hosted future.

---

### 3.2 Компании с распределёнными командами

Примеры:

- retail;
- логистика;
- производство;
- сервисные команды;
- field operations;
- франчайзинговые сети.

Потребности:

- обучать сотрудников в разных точках;
- контролировать обязательные курсы;
- работать с мобильного;
- видеть просрочки.

Преимущество продукта:

- learner-first mobile;
- назначения по группам;
- дедлайны;
- отчёты;
- сертификаты.

---

### 3.3 Компании с compliance-обучением

Потребности:

- обязательное обучение;
- подтверждение прохождения;
- тестирование;
- сертификаты;
- аудит;
- отчётность.

Преимущество продукта:

- assessments;
- certificates;
- audit log;
- completion reports;
- deadline tracking.

---

### 3.4 B2B training providers

Потребности:

- размещать курсы;
- обучать клиентов;
- отслеживать прохождение;
- выдавать сертификаты;
- масштабировать обучение.

Преимущество продукта:

- course management;

- certificates;
- reports;
- branding future.

## 4. Основные боли клиентов

### 4.1 Бизнес-боли

- обучение хаотично хранится в Google Docs/Telegram/Notion;
- невозможно понять, кто прошёл обучение;
- onboarding зависит от конкретных людей;
- нет контроля обязательного обучения;
- нет доказательства прохождения;
- отчёты собираются вручную;
- сотрудники забывают дедлайны;
- enterprise LMS слишком дорогие и сложные.

### 4.2 Пользовательские боли

Учащийся:

- не понимает, что нужно пройти;
- теряет ссылки на материалы;
- не видит прогресс;
- не знает дедлайны.

Админ:

- вручную назначает обучение;
- собирает отчёты вручную;
- не видит статусы;
- не может быстро обновлять курсы.

Менеджер:

- не знает, кто из команды обучен;
- не видит просрочки;
- не может доказать compliance.

## 5. Value Proposition

### 5.1 Для бизнеса

- контроль прохождения;
- снижение ручной работы;

### 5.3 Для администратора

- удобное управление курсами;
- массовые назначения;
- отчёты;
- роли;
- группы;
- audit log.

## 6. MVP commercial package

MVP должен быть продаваем как:

> система для запуска корпоративного обучения, onboarding и контроля прохождения курсов.

Функции MVP для коммерческого использования:

- пользователи и роли;
- группы/департаменты;
- курсы;
- уроки;
- тесты;
- назначения;
- прогресс;
- сертификаты;
- уведомления;
- базовые отчёты;
- audit log;
- Railway deployment;
- Docker portability.

---

## 7. Что не продавать в MVP

Не обещать в MVP:

- AI-персонализацию;
- adaptive learning;
- SCORM/xAPI full support;
- LTI;
- маркетплейс;
- HRIS integrations;
- SSO/SAML;
- white-label multi-tenant enterprise;
- advanced BI;
- offline mobile;
- drag-and-drop course builder уровня Articulate.

Эти функции можно позиционировать как roadmap/future scope.

---

## 8. Дифференциация

### 8.1 Простота против тяжёлых LMS

Позиция:

- не перегруженная enterprise LMS;
- быстрый запуск;
- понятный интерфейс;
- достаточный corporate learning core.

### 8.2 AI-ready, но не AI-зависимая

Позиция:

- продукт проектируется с учётом будущих AI-функций;
- MVP не зависит от AI;
- AI можно добавить позже для генерации курсов, аналитики, рекомендаций, ассистента.

### 8.3 Mobile learner-first

Позиция:

- мобильное приложение ориентировано на учащегося;
- особенно полезно для распределённых команд.

### 8.4 Docker-portable

Позиция:

- можно запустить как SaaS-first;
- в будущем можно разворачивать на сервере клиента;
- не привязано жёстко к одной инфраструктуре.

## 9. Pricing model

### 9.1 Рекомендуемая модель

Основная модель:

- subscription per active user per month;
- тарифы по количеству пользователей;
- отдельные enterprise условия.

### 9.2 Пример тарифов

#### Starter

Для малых команд.

Включает:

- до N пользователей;
- курсы;
- тесты;
- базовые отчёты;
- сертификаты;
- email/support basic.

#### Business

Для компаний среднего размера.

Включает:

- больше пользователей;
- группы/департаменты;
- менеджерские отчёты;
- уведомления;
- расширенные сертификаты;
- API future.

#### Enterprise

Для крупных клиентов.

Включает:

- SSO/SAML;
- advanced audit;
- custom deployment;
- custom branding;
- integrations;
- SLA;
- dedicated support;
- AI features future.

## 12. Demo сценарий

Рекомендуемый demo flow:

1. Войти как Admin.
2. Создать курс.
3. Добавить урок.
4. Добавить тест.
5. Назначить курс группе.
6. Войти как Learner.
7. Пройти урок.
8. Пройти тест.
9. Получить сертификат.
10. Войти как Manager/Admin.
11. Посмотреть отчёт.

Этот demo flow должен быть основой первого продукта.

## 14. Риски коммерческой стратегии

### Риск 1. Слишком широкий scope

Решение:

- продавать конкретный use case: onboarding/compliance.

### Риск 2. Конкуренция с mature LMS

Решение:

- не конкурировать количеством функций;
- конкурировать простотой, скоростью запуска, русским UX, AI-ready подходом.

### Риск 3. Недостаточно доверия

Решение:

- security checklist;
- audit log;
- clear deployment model;
- pilot documentation.

### Риск 4. Недоделанный demo

Решение:

- сначала стабилизировать core learning loop;
- не показывать future scope как готовую функцию.

## 15. Commercial MVP Definition of Done

Продукт можно показывать пилотному клиенту, если:

- Admin может создать курс;
- Learner может пройти курс;
- тест работает;
- прогресс сохраняется;
- сертификат выдаётся;
- отчёт показывает completion;
- роли работают;
- деплой стабилен;
- есть базовая документация;
- нет критических security holes.
