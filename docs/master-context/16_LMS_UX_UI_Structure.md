# 16. LMS UX/UI Structure

Статус: проектный документ для MVP и последующего развития  
Язык продукта: русский  
Контекст: корпоративная LMS, modular monolith first, learner-first mobile, AI-функции вне MVP

---

## 1. Назначение документа

Этот документ описывает UX/UI-структуру LMS: основные пользовательские роли, интерфейсные зоны, экраны, пользовательские сценарии и требования к удобству продукта.

Документ нужен для:

- проектирования frontend-приложения;
- подготовки design system;
- постановки задач frontend AI-агентам;
- согласования экранов с backend API;
- контроля, чтобы MVP не превратился в перегруженную enterprise-систему;
- дальнейшего проектирования мобильного приложения.

---

## 2. Основной принцип UX

Главный принцип продукта:

> LMS должна быть простой для учащегося, управляемой для администратора и достаточно прозрачной для менеджера.

Для MVP важнее:

- понятный вход в обучение;
- быстрый доступ к назначенным курсам;
- прозрачный прогресс;
- простое прохождение уроков и тестов;
- удобная админ-панель для создания и назначения курсов;
- отчёты без перегруженной BI-аналитики.

Не нужно в MVP:

- сложный визуальный конструктор курсов;
- drag-and-drop редактор с десятками блоков;
- маркетплейс курсов;
- социальная лента;
- AI-рекомендации;
- сложная геймификация;
- полноценная HRIS-система.

---

## 3. Роли пользователей и интерфейсные зоны

### 3.1 Learner / Учащийся

Основная роль для прохождения обучения.

Ключевые задачи:

- войти в систему;
- увидеть назначенные курсы;
- продолжить обучение;
- пройти уроки;
- пройти тесты;
- увидеть прогресс;
- скачать или открыть сертификат;
- получить уведомления.

Основные экраны:

- вход;
- личный кабинет;
- мои курсы;
- страница курса;
- проигрыватель урока;
- тест;
- результаты теста;
- сертификаты;
- уведомления;
- профиль.

---

### 3.2 Instructor / Автор или преподаватель

Роль для управления учебным контентом.

Ключевые задачи:

- создавать и редактировать курсы;
- добавлять модули и уроки;
- загружать материалы;
- создавать тесты;
- просматривать результаты учащихся;
- обновлять контент.

Основные экраны:

- список моих курсов;
- создание курса;
- редактирование курса;
- структура курса;
- редактор урока;
- редактор теста;
- результаты по курсу.

В MVP роль Instructor может быть объединена с Admin, если это упрощает реализацию.

---

### 3.3 Manager / Руководитель

Роль для контроля обучения сотрудников своей команды.

Ключевые задачи:

- видеть сотрудников своей группы/департамента;
- отслеживать прогресс;
- видеть просрочки;
- смотреть completion rate;
- выгружать простые отчёты.

Основные экраны:

- dashboard менеджера;
- команда;
- прогресс команды;
- отчёт по курсам;
- отчёт по сотруднику.

---

### 3.4 Admin / Администратор

Главная роль управления LMS.

Ключевые задачи:

- управлять пользователями;
- управлять ролями;
- управлять группами и департаментами;
- управлять курсами;
- назначать обучение;
- смотреть отчёты;
- управлять сертификатами;
- управлять настройками;
- смотреть audit log.

Основные экраны:

- admin dashboard;
- пользователи;
- роли и права;
- группы/департаменты;
- курсы;
- назначения;
- отчёты;
- сертификаты;
- файлы;
- уведомления;
- настройки;
- audit log.

## 5. Learner Portal

### 5.1 Dashboard учащегося

Цель: дать учащемуся быстрый ответ на вопрос: “Что мне нужно пройти сейчас?”

Блоки:

- приветствие;
- продолжить обучение;
- назначенные курсы;
- обязательные курсы;
- дедлайны;
- прогресс;
- последние уведомления;
- сертификаты.

MVP-правило:

- экран должен вести к действию “Продолжить обучение”;
- не перегружать аналитикой;
- не показывать админские термины.

---

### 5.2 Мои курсы

Список курсов, доступных учащемуся.

Фильтры MVP:

- все;
- не начатые;
- в процессе;
- завершённые;
- обязательные.

Карточка курса:

- название;
- краткое описание;
- статус;
- прогресс;
- дедлайн, если есть;
- бейдж обязательности;
- кнопка “Продолжить” или “Начать”.

---

### 5.3 Страница курса

Содержит:

- название;
- описание;
- автора/организацию;
- длительность;
- список модулей и уроков;
- прогресс;
- требования завершения;
- кнопка начала/продолжения.

MVP:

- линейная структура курса;
- уроки идут по порядку;
- можно разрешить или запретить свободную навигацию настройкой курса.

---

### 5.4 Lesson Player

Один из самых важных экранов LMS.

Содержит:

- заголовок курса;
- заголовок урока;
- основной контент урока;
- вложенные материалы;
- прогресс;
- навигация назад/вперёд;
- кнопка “Отметить как пройдено”;
- переход к тесту, если он нужен.

Поддерживаемый контент MVP:

- текст;
- видео-ссылка или встроенное видео;
- PDF/файл;
- изображение;
- базовые вложения.

Не включать в MVP:

- SCORM player;
- xAPI tracking;
- интерактивные симуляции;
- AI-тьютор в уроке.

---

### 5.5 Assessment / Тест

MVP-структура:

- intro screen;
- список вопросов по одному или все сразу;
- single choice;
- multiple choice;
- результат;
- pass/fail;
- количество попыток;
- сохранение результата.

Экран результата:

- набранный процент;
- проходной процент;
- статус;
- возможность повторной попытки, если разрешено;
- переход к сертификату, если курс завершён.

---

### 5.6 Сертификаты

Экран содержит:

- список полученных сертификатов;
- курс;
- дата выдачи;
- срок действия, если есть;
- статус;
- кнопка просмотра/скачивания.

MVP:

- простой PDF/HTML-сертификат;
- уникальный идентификатор сертификата;
- связь с пользователем и курсом.

---

## 6. Admin Portal

### 6.1 Admin Dashboard

Цель: дать администратору обзор состояния обучения.

Блоки:

- количество пользователей;
- активные курсы;
- назначенные курсы;
- completion rate;
- просроченные назначения;
- последние действия;
- быстрые действия.

MVP-быстрые действия:

- создать курс;
- добавить пользователя;
- назначить курс;
- открыть отчёты.

---

### 6.2 Пользователи

Функции MVP:

- список пользователей;
- поиск;
- фильтр по роли;
- фильтр по статусу;
- создание пользователя;
- редактирование;
- деактивация;
- назначение роли;
- привязка к группе/департаменту.

Поля:

- имя;
- email;
- роль;
- статус;
- организация;
- департамент;
- дата создания;
- последняя активность.

---

### 6.3 Роли и права

MVP-роли:

- Admin;
- Instructor;
- Manager;
- Learner.

В MVP не нужен сложный visual permission builder.

Достаточно:

- фиксированные роли;
- RBAC на backend;
- UI скрывает недоступные действия;
- backend всегда проверяет права повторно.

---

### 6.4 Группы и департаменты

Функции:

- создать группу;
- создать департамент;
- добавить пользователей;
- назначить курс группе;
- посмотреть прогресс группы.

Для MVP можно реализовать одну универсальную сущность `group`, если департаменты усложняют модель.

---

### 6.5 Курсы

Админский список курсов.

Функции:

- список;
- поиск;
- статус draft/published/archived;
- создание;
- редактирование;
- публикация;
- архивирование;
- просмотр назначений;
- просмотр результатов.

---

### 6.6 Course Editor

MVP-редактор курса:

- основные данные курса;
- модули;
- уроки;
- материалы;
- тест;
- настройки завершения;
- публикация.

Не делать в MVP:

- сложный visual builder;
- branching logic;
- adaptive content;
- SCORM/xAPI;
- marketplace blocks.

---

### 6.7 Назначения

Экран назначения курсов.

Функции:

- выбрать курс;
- выбрать пользователей/группу;
- указать обязательность;
- указать дедлайн;
- создать назначение;
- смотреть статус выполнения.

---

### 6.8 Отчёты

MVP-отчёты:

- прогресс по пользователям;
- прогресс по курсам;
- завершённые/незавершённые назначения;
- просрочки;
- сертификаты.

Форматы:

- таблица в интерфейсе;
- CSV export как future или MVP+, если быстро реализуется.

---

### 6.9 Audit Log

Внутренний журнал действий.

Показывает:

- кто сделал действие;
- что изменил;
- когда;
- объект;
- тип события.

MVP-события:

- login;
- user created;
- role changed;
- course created;
- course published;
- assignment created;
- assessment submitted;
- certificate issued.

## 8. Instructor Portal

В MVP может быть доступен только Admin/Instructor как единый административный интерфейс.

Если выделяется отдельно, Instructor видит:

- свои курсы;
- редактор курса;
- результаты по своим курсам;
- комментарии/обратную связь, если предусмотрено.

## 9. Навигация

### 9.1 Desktop

Рекомендуемая структура:

- верхняя панель: профиль, уведомления, организация;
- боковое меню: разделы по роли;
- рабочая область: таблицы, карточки, формы;
- breadcrumbs для вложенных экранов.

### 9.2 Mobile Web

Для learner-портала:

- bottom navigation;
- dashboard;
- мои курсы;
- уведомления;
- профиль.

Admin mobile не является приоритетом MVP.

Нужны:

- Button;
- Input;

- Radio;
- Modal;
- Drawer;

- Card;
- Badge;
- Progress bar;

### 10.2 Статусы

Общие статусы:

- archived;
- assigned;
- not_started;

- нет пользователей;
- нет сертификатов;
- нет отчётов.

## 11. UX-правила для AI-агентов

AI-агент при создании frontend должен соблюдать:

1. Не добавлять функции вне MVP без отдельной задачи.
2. Не делать сложный UI там, где достаточно таблицы и формы.
3. Все формы должны иметь валидацию.
4. Все действия изменения должны иметь подтверждение или понятную обратную связь.
5. Ошибки API должны отображаться пользователю.
6. Ролевой доступ должен проверяться не только в UI, но и на backend.
7. Тексты интерфейса должны быть на русском языке.
8. Не использовать технические термины в learner-интерфейсе.
9. Не смешивать admin UX и learner UX.
10. Mobile-first применять прежде всего к learner flow.

## 12. MVP UX Scope

В MVP обязательно:

- login;
- learner dashboard;
- my courses;
- course page;
- lesson player;
- assessment;
- certificates;
- admin dashboard;
- users;
- courses;
- course editor;
- assignments;
- basic reports;
- notifications list;
- profile.

В MVP можно отложить:

- advanced analytics;
- drag-and-drop builder;
- social learning;
- AI recommendations;
- SCORM/xAPI player;
- HRIS integrations;
- custom branding per tenant;
- offline mobile mode.

---

## 13. Definition of Done для UX/UI

Экран считается готовым, если:

- есть desktop layout;
- learner-critical screens адаптированы под mobile;
- подключены реальные API или mock только временно обозначен;
- есть loading/error/empty states;
- есть валидация форм;
- учтены роли;
- тексты на русском;
- нет функций вне MVP;
- есть smoke test основного сценария.
