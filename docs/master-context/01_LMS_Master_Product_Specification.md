# 01. LMS Master Product Specification

**Проект:** корпоративная LMS / Learning Operating System  
**Язык продукта:** русский, английский, казахский, китайский, в перспективе мульти зычность.   
**Формат документа:** базовая продуктовая спецификация для разработки MVP  
**Основное назначение:** зафиксировать, какой продукт создаётся, для кого, с какими границами MVP и какими принципами развития.

---

## 1. Статус документа

Этот документ является **главной продуктовой основой проекта LMS**. Он должен использоваться как первичный ориентир перед проектированием базы данных, API, backlog, GitHub Issues, инструкций для AI coding agents и плана разработки.

Документ объединяет ключевые решения из текущих master/spec/brief-файлов проекта и приводит их к единому виду.

## 2. Краткое описание продукта

**FACT:** создаётся корпоративная LMS-платформа для B2B-сегмента.

Продукт должен позволять компаниям управлять обучением сотрудников: создавать и редактировать курсы, назначать обучение группам и пользователям, создавать тесты с таймингом прохождения и возможность отображение правильных ответов, создавать траектории обучения, отслеживать прогресс, проводить тестирование, проводить опросы, проводить пульс - опросы, проводить контрольный срез знаний, выдавать сертификаты и получать отчётность, проводить полевые опросы по чек - листам, создавать и оргструктуру редактировать, публиковать новости, проводить смешанное обучение, проводить практические занятия по чек листам, экспортировать и импортировать курсы в SCORM формате, экспортировать и импортировать пользователей по шаблону Excel и тд, создавать и редактировать настраиваемые пользовательские группы, настраивать пользовательский интерфейс, создавать и выдавать права (Администратор аккаунта, администратор, обучаемый, инструктор, куратор, руководитель), позволять кастомизировать страницы LMS - платформы с Администратора аккаунта.

Продукт должен развиваться по принципу:

> сначала практичный MVP на modular monolith, затем коммерческая B2B-версия, затем enterprise-ready расширение и только после этого AI-first функции.

## 3. Главная цель продукта

Создать LMS, которую можно использовать для:

- адаптации новых сотрудников;
- обязательного compliance-обучения;
- внутреннего корпоративного обучения;
- повышения квалификации сотрудников;
- проверки знаний;
- геймификации;
- проведение практических занятий;
- выдачи сертификатов;
- контроля обучения руководителями, администраторами, кураторами, инструкторами;
- подготовки к будущему AI-assisted learning.

## 6. Главный MVP learning loop

MVP должен закрыть минимальный, но полноценный цикл корпоративного обучения:

1. Администратор настраивает систему, роли, права, кастомизацию.
2. Администратор создаёт и управляет пользователями, группами.
3. Администратор создает оргструктутру.
4. Администратор создает и назначает траектории обучения, настраивает автоматическое назначение курсов и траекторий обучения.
5. Instructor/Инструктор или куратор создаёт курс, траектории обучения.
6. Instructor/Инструктор или куратор назначает курс, траекторию обучения пользователю или группе.
4. Learner видит назначенный курс или траекторию.
5. Learner проходит курс или траекторию.
6. Система фиксирует прогресс.
7. Learner сдает тест.
8. Система сохраняет результат.
9. При выполнении условий Learner получает сертификат или баллы.
10. Администратор, Руководитель, Инструктор или Куратор видит отчёты по прохождению.

Этот цикл является ядром продукта. Всё, что не помогает этому циклу, не должно попадать в MVP без отдельного обоснования.

## 8. Не входит в MVP

**OUT OF MVP:**

- полноценный AI tutor;
- RAG по корпоративным документам;
- Julia analytics service;
- сложная adaptive learning логика;
- микросервисная архитектура;
- multi-region deployment;
- marketplace курсов;
- SCORM/xAPI/LTI как полноценная реализация;
- offline-first mobile app;
- сложная BI-платформа;
- white-label enterprise customization;
- advanced HRIS integrations;
- deep SSO/SAML enterprise pack;
- сложная система оплаты и биллинга.

Эти функции могут быть включены в future scope, но не должны блокировать MVP.

## 9. Ключевые продуктовые принципы

### 9.1 Простота MVP

MVP должен быть достаточно простым, чтобы его мог разработать один человек с помощью AI-агентов, но достаточно цельным, чтобы показать реальную ценность клиенту.

### 9.2 Corporate-first

Фокус не на массовом B2C-обучении, а на корпоративном обучении, назначениях, контроле и отчётности.

### 9.3 Modular monolith first

Система должна начинаться как модульный монолит с понятными границами доменов. Микросервисы допустимы только после появления реальных причин.

### 9.4 AI future scope

AI-функции должны проектироваться как будущий слой, но не включаться в MVP как обязательная часть.

### 9.5 Mobile learner-first

Мобильное направление важно, но в MVP основной приоритет — responsive learner experience. Нативное мобильное приложение может развиваться после стабилизации core LMS.

2. Видит список сотрудников.

## 13. Технологические решения верхнего уровня

**FACT:** для проекта зафиксированы следующие решения:

- backend: TypeScript;
- frontend: React + TypeScript;
- database: PostgreSQL;
- storage: S3-compatible;
- deployment: Railway-first;
- portability: Docker-first;
- architecture: modular monolith first;
- repository: private GitHub;
- development mode: один человек + AI-агенты GPT/Claude;
- product language: Russian-first;
- AI features: future scope, not MVP.

MVP можно считать успешным, если:

- Admin может создать пользователей и группы;
- Admin может создать организацию или использует существующий tenant;
- Admin может создать права, роли, группы, траектории;
- Admin может увидеть отчёт;
- Instructor/Инструктор или куратор может создать и опубликовать курс;
- Instructor/Инструктор или куратор может назначить курс пользователю или группе;

## 15. Главные риски проекта

### 15.1 Переусложнение MVP

Риск: попытка добавить enterprise, AI, SCORM, mobile offline, microservices и сложную аналитику до готовности core LMS.

Митигировать через строгий MVP scope.

### 15.2 Несогласованность документов

Риск: backlog, API и database model будут описывать разные сущности.

Митигировать через поэтапную подготовку документации.

### 15.3 Слабая роль AI-агента

Риск: AI-агент будет писать код без единого контекста.

Митигировать через отдельные инструкции для AI coding agent.

### 15.4 Недостаточная безопасность

Риск: корпоративная LMS требует контроля доступа, audit log и защиты данных.

Митигировать через Security Checklist и RBAC с самого начала.
