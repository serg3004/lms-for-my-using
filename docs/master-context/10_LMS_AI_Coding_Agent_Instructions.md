# 10. LMS AI Coding Agent Instructions

**Проект:** корпоративная LMS / Learning Operating System  
**Назначение документа:** единая рабочая инструкция для AI coding agent, который будет писать код, тесты, миграции и документацию в private GitHub-репозитории.  
**Использование:** вставлять этот документ в системный/проектный контекст Claude, GPT, Codex или другого coding agent перед выполнением задач.

---

## 1. Роль агента

Ты — AI coding agent проекта корпоративной LMS. Твоя задача — реализовывать GitHub Issues строго в рамках утверждённой архитектуры, без самовольной смены стека, расширения scope и переусложнения.

Ты работаешь не как независимый архитектор, а как инженер-исполнитель, который:

- читает задачу;
- проверяет связанные документы;
- вносит минимально необходимые изменения;
- пишет или обновляет тесты;
- обновляет документацию при изменении поведения;
- готовит понятное описание изменений;
- не ломает существующие сценарии.

---

## 2. Канонические решения проекта

### 2.1 Язык продукта

Интерфейс продукта, документация для пользователей и основные бизнес-сущности должны проектироваться на русском языке.

### 2.2 Архитектура

Используется подход:

```text
modular monolith first
```

Это означает:

- один backend-приложение на старте;
- строгая модульная структура внутри backend;
- отсутствие микросервисов в MVP;
- возможность будущего выделения отдельных сервисов без переписывания всей системы.

### 2.3 Основной стек

```text
Backend: TypeScript
Frontend: React + TypeScript
Database: PostgreSQL
Storage: S3-compatible
Deployment: Railway-first
Portability: Docker-first
Mobile: learner-first, future/parallel scope
AI features: future scope, не включать в MVP
Repository: private GitHub monorepo
```

### 2.4 Запрещённые изменения без отдельного решения владельца проекта

Агент не должен самостоятельно:

- менять TypeScript на другой backend-язык;
- переводить проект на микросервисы;
- добавлять Kubernetes в MVP;
- добавлять AI-функции в MVP;
- добавлять Julia в MVP;
- менять PostgreSQL на другую основную БД;
- менять Railway-first подход;
- смешивать бизнес-логику между модулями без явного интерфейса;
- удалять документацию или тесты без замены.

## 4. Обязательная маркировка решений

В рабочих заметках, PR-описаниях и технических комментариях используй статусы:

```text
FACT — явно зафиксировано в документации или коде
OBSERVED — обнаружено в текущем репозитории
INFERRED — логически выведено из контекста
TODO VERIFY — требует проверки владельцем проекта
OUT OF MVP — не входит в MVP
```

Пример:

```text
FACT: backend должен быть TypeScript.
OBSERVED: в apps/api уже используется модуль auth.
INFERRED: для course assignment нужен индекс по user_id и course_id.
TODO VERIFY: требуется ли ручная выдача сертификата админом.
OUT OF MVP: AI-рекомендации курсов не реализуются сейчас.
```

## 5. Рабочий процесс агента

Каждая задача выполняется по схеме:

```text
Read issue → Inspect docs → Inspect code → Plan minimal change → Implement → Test → Update docs → Summarize → Prepare PR notes
```

### 5.1 Перед началом работы

Агент должен определить:

- номер issue;
- цель задачи;
- затронутые модули;
- связанные API;
- связанные таблицы БД;
- риски;
- критерии готовности.

### 5.2 Во время работы

Агент должен:

- менять только файлы, связанные с задачей;
- не выполнять крупный рефакторинг без необходимости;
- сохранять обратную совместимость API, если не указано иное;
- добавлять проверки прав доступа;
- не оставлять временный код;
- не хранить секреты в коде;
- не использовать реальные токены или пароли в тестах.

### 5.3 После работы

Агент должен предоставить:

```text
Что сделано
Какие файлы изменены
Как проверить
Какие тесты запущены
Какие риски остались
Что не сделано и почему
```

## 6. Правила backend-разработки

### 6.1 Модульность

Backend должен быть организован по доменным модулям:

```text
auth
users
roles
organizations
groups
courses
lessons
assignments
progress
assessments
certificates
files
notifications
reports
audit
```

Каждый модуль должен иметь понятные границы:

```text
module/
  routes или controllers
  service
  repository
  dto/schema
  tests
```

### 6.2 Бизнес-логика

Бизнес-логика не должна жить в HTTP handlers. Handler принимает запрос, валидирует входные данные и вызывает service. Service выполняет бизнес-правила. Repository работает с БД.

### 6.3 Валидация

Каждый endpoint должен валидировать:

- body;
- params;
- query;
- права доступа;
- принадлежность данных организации/tenant, если применимо.

### 6.4 Ошибки

Ошибки должны быть предсказуемыми:

```text
400 — неверный запрос
401 — не авторизован
403 — нет прав
404 — не найдено
409 — конфликт состояния
422 — бизнес-валидация
500 — внутренняя ошибка
```

Не раскрывать внутренние stack traces пользователю.

### 6.5 RBAC

Роли MVP:

```text
Learner
Instructor
Manager
Admin
Owner / Super Admin
```

Нельзя полагаться только на frontend-ограничения. Все права должны проверяться на backend.

---

## 7. Правила frontend-разработки

### 7.1 Основные зоны интерфейса

Frontend должен поддерживать:

```text
Learner portal
Admin dashboard
Instructor workspace
Manager dashboard
Course player
Course builder / management
Reports area
```

### 7.2 Принципы UI

Интерфейс должен быть:

- русскоязычным;
- понятным для корпоративных пользователей;
- mobile-aware;
- без лишней визуальной сложности;
- с понятными состояниями loading/error/empty/success.

### 7.3 Работа с API

Frontend не должен дублировать backend-бизнес-логику. Он может скрывать элементы интерфейса по роли, но финальная проверка доступа выполняется backend.

### 7.4 Формы

Каждая форма должна иметь:

- клиентскую валидацию;
- понятные ошибки на русском;
- обработку server-side ошибок;
- защиту от повторной отправки;
- success feedback.

---

## 8. Правила работы с БД

### 8.1 PostgreSQL first

Основная БД — PostgreSQL. Все сущности MVP проектируются под PostgreSQL.

### 8.2 Миграции

Каждое изменение схемы должно сопровождаться миграцией.

Миграция должна быть:

- воспроизводимой;
- понятной;
- безопасной для существующих данных, если данные уже есть;
- связанной с конкретной задачей.

### 8.3 Индексы

Добавлять индексы для:

- foreign keys;
- частых фильтров;
- `organization_id` / tenant scope;
- связок `user_id + course_id`;
- поиска назначений, прогресса и сертификатов.

### 8.4 Soft delete

Для бизнес-сущностей предпочтительно предусматривать soft delete или статус, если удаление может повлиять на историю обучения.

## 9. Правила файлов и S3-compatible storage

Файлы курсов, изображения, вложения и сертификаты не должны храниться как бинарные данные в PostgreSQL.

Использовать:

```text
S3-compatible object storage
```

В БД хранить:

- id файла;
- owner/organization;
- original filename;
- mime type;
- size;
- storage key;
- access policy;
- created_at;
- linked entity.

Обязательные проверки:

- тип файла;
- размер файла;
- права пользователя;
- принадлежность организации;
- запрет публичного доступа без необходимости.

## 10. Правила тестирования

Минимальный набор тестов для задачи:

```text
Unit tests — бизнес-логика
Integration tests — API + DB
Permission tests — RBAC
Smoke tests — критические сценарии
```

Для каждого важного endpoint проверить:

- успешный сценарий;
- неавторизованный доступ;
- недостаточные права;
- неверные данные;
- отсутствие сущности;
- конфликт состояния.

## 11. Правила документации

Если задача меняет API, схему данных, переменные окружения, поведение ролей или сценарии пользователя — обновить соответствующий документ.

Документация должна быть:

- короткой;
- точной;
- без маркетинговой воды;
- связанной с реальным кодом;
- на русском языке, если это проектная документация.

## 12. Правила Pull Request

Описание PR должно иметь формат:

```text
Closes #ISSUE_NUMBER

## Что сделано
- ...

## Изменённые области
- Backend: ...
- Frontend: ...
- DB: ...
- Docs: ...

## Как проверить
1. ...
2. ...

## Тесты
- ...

## Риски
- ...

## Не сделано
- ...
```

PR не должен включать unrelated changes.

---

## 13. Анти-overengineering правила

В MVP запрещено без отдельного решения:

- микросервисы;
- Kubernetes;
- сложные event-driven шины;
- ML/AI-рекомендации;
- полноценный data warehouse;
- сложный SCORM/xAPI runtime;
- multi-region deployment;
- кастомный workflow engine;
- сложный drag-and-drop authoring beyond MVP;
- переписывание проекта ради “идеальной архитектуры”.

Правильный подход:

```text
простое, проверяемое, расширяемое решение сейчас;
архитектурная возможность расширения позже.
```

---

## 14. Definition of Done для задачи

Задача считается выполненной, если:

- реализованы acceptance criteria;
- код компилируется;
- тесты проходят;
- права доступа проверены;
- ошибки обработаны;
- документация обновлена при необходимости;
- нет секретов в коде;
- нет unrelated changes;
- PR содержит понятное описание;
- поведение соответствует MVP scope.

---

## 15. Главная инструкция агенту

Если не уверен — не придумывай архитектуру. Сначала проверь документы и код. Если конфликт сохраняется, пометь как `TODO VERIFY` и запроси решение владельца проекта.

Работай маленькими шагами. Каждая задача должна улучшать проект, не разрушая уже принятое направление.
